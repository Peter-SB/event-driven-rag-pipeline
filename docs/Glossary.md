
# Glossary

This file is intended to act as a glossary of key terms and concepts used throughout the codebase and documentation. It is not meant to be exhaustive, but rather to provide clarity on the most important and commonly used terms.

# Post
A "post" refers to a piece of content that is being processed by the system. While document would also be aptly named, "post" is use for this projects integration to my Reddit Bookmark App. However I intend for this to not be the only source of content.

# Chunk
A "chunk" is a smaller piece of data representing some text that has will be embedded and stored in the vector database. A chunk is always post body, it could also be a post summary, or even new data generated by inference. The important part is that a chunk is the unit of data that gets embedded and stored in the vector database. It could have been called "embedding" or "vector", however I felt that was potentially confusing since the chunk itself is not the embedding, but rather the input to the embedding process.

# Event Log (Kafka/Redpanda/Postgres Mock)
The "event log" is the append-only log of events that represents the state changes and actions in the system. In production this is implemented with Kafka/Redpanda, but for homelab use there is a Postgres-backed mock abstraction implementation. The event log is immutable, replayable, and consumer-agnostic, meaning that it can be consumed by multiple different consumers without affecting the producers or other consumers.

**Notes:**
- The Event Log does not store state, it stores events
- On ordering: Kafka provides ordering guarantees only within partitions, so events that are related and need to be processed in order should be sent to the same partition. This is typically done by using a key that represents the entity or context of the event (e.g post_id) when producing the event.
- Though events are immutable, they do have retention policies and will only be retained for so long.

# RabbitMQ

**Exchange** "An exchange is a message routing agent that receives messages from producers and routes them to queues based on routing rules."

**Binding** "A binding is a "link" that you set up to bind a queue to an exchange."

**Routing Key** is a message attribute that the exchange uses to determine how to route the message to queues

**Queue** "A queue is a buffer that stores messages until they are consumed by a consumer.

### Task Message
A "task message" is the payload that is sent to RabbitMQ queues for workers to consume.

### Worker
processes messages from a RabbitMQ queue with custom logic for resource management.

### Handler
The function that contains the logic for processing a specific type of task message.
